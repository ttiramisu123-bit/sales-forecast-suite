一、整体用途

工作台分为两个独立系统：先用 MSKU 销售预测系统生成“销售侧预测”，再用 SKU 需求预测系统把 MSKU 预测拆解到真实 SKU 需求，并通过风险平衡降低牛鞭效应。

MSKU 销售预测系统 面向销售表现，按 MSKU、Type、价格、历史销量、季节性和人工目标生成未来 12 个月最终预测销量与销售额。
SKU 需求预测系统 面向供应需求，把 MSKU 最终预测按映射关系转成 SKU 需求，再结合风险评分、销售状态和人工修正得到 SKU 人工终版。
两套系统独立运行 MSKU 负责预测输入，SKU 负责需求拆解和风险平衡。两边不混在一个页面里，避免数据量过大导致卡顿。
核心原则：MSKU 预测看销售趋势，SKU 预测看真实需求和供应风险。运营调整应先确认销售预测，再进入 SKU 维度检查映射、迭代、风险和终版需求。

二、标准操作流程

Step 1进入 MSKU 系统，导入历史销量基础数据。
Step 2检查预测、目标涨幅、Type 汇总和趋势曲线。
Step 3确认后保存 SKU 输入包到云端。
Step 4进入 SKU 系统，导入最新 MSKU 预测包。
Step 5查看风险、来源、终版需求，必要时人工调整并导出。

MSKU 系统操作顺序

  1. 导入历史销量数据，确认 SKU 数、有效 SKU、价格、Type、历史月份读取正常。
  2. 查看月度销售额趋势、Type/类目汇总、趋势曲线和 SKU 预测明细，确认整体方向是否符合业务预期。
  3. 如需要设置目标增长，先在“月度目标分配”填写全局目标，点击“全局应用”生成 Type 建议。
  4. 确认 Type 建议后，点击“类目应用”，才会把 Type 涨幅写入 SKU 最终预测。
  5. 若个别 MSKU 需要特殊处理，可在 SKU 预测明细里编辑 12 个月终版预测并填写调整原因。
  6. 确认无误后，点击“保存 SKU输入包云端”，供 SKU 系统读取。

SKU 系统操作顺序

  1. 点击“导入最新MSKU预测包”，系统会读取 MSKU 端保存的最新 SKU 输入包。
  2. 系统自动加载内置 MSKU-SKU 映射关系和 SKU 销售状态表。
  3. 点击“生成SKU预测”，系统会检查缺映射、停售/清库缺迭代、迭代循环等问题。
  4. 生成成功后，先看数据诊断和异常 SKU 看板，再查看 SKU 预测总表。
  5. 点击 SKU 行的“详情/来源明细”，查看来源 MSKU 数、来源 Type、映射需求、风险和平衡系数。
  6. 如确需人工调整，点击编辑后修改任意月份的人工终版需求，填写备注并保存。

1. 基础预测链路

步骤说明运营关注点

异常清洗对明显异常高值、异常低值和突然归零进行保守处理，避免单月极端值拉飞预测。关注 3 月旺季等真实高峰，不要把真实旺季误判为异常。

目标校准通过全局目标和 Type 目标调整未来月份，但必须点击应用才生效。未点击应用前只是建议或待生效值。
人工终版最终预测 = 公式预测 + 目标校准 + 活动/策略/人工修正。终版才是给后续 SKU 系统使用的口径。

2. 核心公式口径

清洗后销量 → 内部基准 rawBaseForecast → Type+销量分层校准 → 近期刹车 + 增长奖励 + 防过冲/尾部收缩 → 公式预测 公式预测 × Type目标系数 + 活动加量/人工直改 = MSKU最终预测 MSKU最终预测 × 价格 = 预估销售额

3. 内部口径：历史销量清洗规则

系统会先把历史销量转成非负数，再用正销量的中位数作为基准值。如果历史月份不足 3 个月，或没有正销量，则不做异常替换。

清洗项触发条件替换方式影响
异常高值本月销量同时高于“基准值 × 阈值”和“相邻月份均值 × 阈值”。替换为左右相邻月份均值；没有相邻值时使用基准值。避免单月大促、异常订单或录入错误把后续预测拉飞。
低位归零本月为 0，但相邻月份均值高于基准值 50%，且相邻均值至少为 3。替换为左右相邻月份均值。避免断货、漏记或短暂下架导致预测被压得过低。
正常旺季如果高峰月份没有同时超过基准和相邻月份阈值，则不会替换。保留原始销量。例如 3 月旺季如果周边月份也有趋势支撑，会尽量保留。

4. 内部口径：MSKU 公式预测各影响项

影响项计算方式作用方向运营解释
清洗后均值清洗后正销量均值。决定预测底盘。均值越高，后续预测整体越高。
稳定层级有效月份少于 3 为样本不足；CV ≤ 0.35 为高稳定；CV ≤ 0.8 为中稳定；其余为低稳定。决定 SKU 自身趋势、Type 趋势、SI 的权重。越稳定越相信 SKU 自己，越不稳定越多参考 Type 趋势且限制增长。
SKU 趋势近 6 个月均值 / 前 6 个月均值，不足 12 个月时按半窗口；限制在 0.90 - 1.12。上升会抬高基准，下降会压低基准。避免历史趋势过度外推。
Type 趋势同 Type 清洗销量汇总后计算趋势。让 SKU 跟着品线大方向走。防止单个 SKU 历史异常导致和品线趋势背离太大。
混合趋势SKU趋势 × 稳定层级SKU权重 + Type趋势 × 稳定层级Type权重。形成趋势后均值。高稳定更看自己，低稳定/样本不足更看 Type。
SI 季节性季节折算 = 1 + (原始SI - 1) × 稳定层级SI权重。旺季放大，淡季收缩。稳定 SKU 更能吃到季节性，低稳定 SKU 的季节性会打折。
月环比封顶预测月相对上月增长受稳定层级 up/peak 参数限制；下降受 down 参数限制。防止预测突然跳涨或跳跌。旺季允许更高增长，普通月份更保守。
历史月增量封顶上涨时还会参考历史平均正增长量。限制“环比百分比”带来的过度增长。小基数 SKU 不会因为百分比涨幅过大而被拉飞。
销量分层按清洗后历史总销量累计贡献分为头部、中部、尾部；头部内部再识别超级头部。决定后续校准、机会和保护强度。头部/中部给机会，尾部更保守。
滚动回测校准用历史最后 2 个月做验证月，按 Type + 销量分层计算 实际销量 / 内部基准预测。修正不同类目和分层的系统性高估/低估。不使用未来实际值；只是用已发生历史月份做自校准。
近期加权刹车最近 4 个月按 0.4、0.3、0.2、0.1 加权；当近期弱于历史均值时压低预测。只负责刹车，不直接放大。防止近 2-4 月已经明显变弱的 SKU 继续按旧高峰预测。
头/中部增长奖励当近期加权销量 / 历史月均 ≥ 1.30 时，头部给小幅奖励，中部给略高奖励。给增量来源更多机会。用于捕捉近期增长，但幅度受后续防过冲控制。
头部防过冲如果头部 SKU 的内部基准已经接近期销量，则超级头部最多上浮约 5%，普通头部最多上浮约 8%。防止大基数 SKU 因小百分比变成大绝对增量。头部 SKU 基数大，保护更强。
中部防过冲如果中部 SKU 的内部基准已接近期销量，则限制额外上浮空间。避免中部被短期趋势过度拉高。中部保留机会，但不无限放大。
尾部低动销收缩尾部、历史小体量或近 2 月无动销时，对预测设置更低上限；完全无动销且无历史销量可归零。强压尾部/零动销预测。减少尾部 SKU 误备和牛鞭效应风险。

5. 内部口径：优化公式预测完整链路

当前系统展示的“公式预测”已经不是旧基础公式，而是优化后的新公式预测。旧基础公式只作为内部基准 rawBaseForecast 参与计算，不再单独展示、不再作为导出口径。

环节公式/规则影响方向说明
内部基准rawBaseForecast = 清洗后均值 × 混合趋势 × SI季节性，并受月环比和历史增量封顶。形成原始预测底座。这是新算法底座，不直接给运营看。
销量分层按历史总销量累计贡献：前 50% 为头部，50%-80% 为中部，其余为尾部；头部 Top 10% 或月均较高为超级头部。决定机会/保护/收缩策略。让不同体量 SKU 使用不同风险口径。
滚动校准calibration = 历史最后 2 个月实际销量 / 同口径回测预测，按 Type+分层聚合，限制在 0.85-1.35。修正类目和分层偏差。如果某 Type+分层历史一直低估，会适度上调；一直高估会适度下调。
机会系数头部/中部基础机会系数约 1.05，尾部 1.00。给头部和中部更多增长空间。符合“增量主要来自头部/中部”的业务判断。
近期刹车recentWeighted = 最近1月×0.4 + 最近2月×0.3 + 最近3月×0.2 + 最近4月×0.1；recentRatio = recentWeighted / 历史月均。近期弱则压低。近期权重本身不负责放大，避免短期噪声推高预测。
增长奖励当 recentRatio ≥ 1.30 且近期有销量时，头部约 ×1.05，中部约 ×1.08，尾部不奖励。捕捉真实增量。这是选择“近期刹车 + 增长奖励版”的核心差异。
防过冲头部若 rawBaseForecast 已接近期加权销量，则超级头部最多约 +5%，普通头部最多约 +8%；中部也限制额外上浮。控制大基数绝对增量。避免预测总量看起来合理，但单品过冲。
尾部收缩尾部低历史总量、近 2 月无动销或完全无动销时设置更低上限。更保守。尾部的目标是降低误备和库存风险。
公式预测 = rawBaseForecast × Type+分层校准系数 × 头/中部机会系数 × 近期刹车系数 × 头/中部增长奖励系数 再经过头部/中部防过冲、尾部低动销收缩后，得到页面展示和导出的“公式预测”。 历史预测版本只用于曲线对比和参考，不参与公式预测计算。

6. 月度目标分配

月度目标分配分为两层：先从全局目标拆到 Type 建议，再从 Type 目标写入 SKU 最终预测。这样可以避免输入一个全局涨幅后立刻重算几千条 SKU，降低卡顿和误操作风险。

按钮作用影响范围
全局应用按 Type 稳定度和趋势，把全局涨幅分配成 Type 建议涨幅。只影响 Type 建议值，不直接改 SKU。
类目应用把 Type 最终涨幅写入对应 SKU 的预测。刷新 SKU 明细、KPI、趋势条和销售额汇总。
清空人工覆盖清除 Type 人工覆盖值。需要再次点击类目应用才会影响最终预测。
Type调整倾向 T = 稳定度因子 × 趋势因子 稳定度因子：高稳定 0.8，中稳定 1.0，低稳定 1.3，样本不足 0.6 趋势因子：近3月/前3月 > 1.15 为 1.3；1.00-1.15 为 1.0；0.85-1.00 为 0.8；< 0.85 为 0.5 Type建议涨幅 = 全局涨幅 × 当前Type T / 所有Type T均值 最终涨幅限制 = -5% 到 +25%；目标系数限制 = 0.3 到 2.0

Type 涨幅写入 SKU 时，还会做轻量 SKU 稳定度微调，避免同一 Type 下所有 SKU 完全同涨同跌。

SKU微调因子 = 1 + 0.15 × (SKU稳定度因子 - Type加权稳定度因子) 微调范围限制在 0.93 - 1.07，并按销售额加权校准，保证 Type 总量基本不被破坏。

四、SKU 预测口径

1. 三个版本的含义

版本怎么得到用途
映射版本MSKU 最终预测 × 映射用量，按最终 SKU 聚合。表示销售预测自然传导出来的 SKU 需求。
系统平衡版映射版本 × 风险平衡系数。用于抑制牛鞭效应，避免高风险 SKU 需求被过度放大。
人工终版默认等于系统平衡版；有人工系数、加减量或直改后，以人工规则为准。运营确认后的最终 SKU 需求。

2. 映射与迭代逻辑

SKU映射需求 = SUM(MSKU最终预测 × 映射用量) 如果多个 MSKU 指向同一个 SKU，会自动聚合。 如果 SKU 状态为清库/停售且有迭代 SKU，会把需求转入迭代 SKU 后再聚合。 如果清库/停售 SKU 有预测需求但没有迭代 SKU，系统会阻止生成并提示补全。
映射表缺失或停售/清库缺迭代，会直接影响 SKU 需求结果。遇到红色诊断时，优先补映射关系或 SKU 状态表，不建议绕过。

3. 风险评分与平衡系数

SKU 系统会根据来源复杂度、稳定性、人工调整、增长幅度、活动/清仓和迭代转入情况计算风险分。风险越高，系统平衡系数越低。

风险来源触发逻辑可能加分
来源 MSKU 过多来源 MSKU 达到 4、9、20 个以上。约 +10 / +20 / +30
跨 Type 来源同一 SKU 来源跨 2 个或 4 个以上 Type。约 +8 / +15
低稳定来源占比高低稳定或样本不足来源占比达到 20% 或 50%。约 +10 / +20
人工调整来源占比高来源 MSKU 中已有人工调整占比较高。约 +10 / +20
月度增长过快映射需求环比增长超过 30% 或 60%。约 +10 / +20
活动/清仓来源来源 MSKU 带活动或清仓标记。约 +10
迭代转入清库/停售 SKU 的需求转入当前 SKU。约 +8 / +12

4. 内部口径：SKU 风险分完整定义

SKU 风险分不是销售好坏评分，而是“需求被 MSKU 预测传导后是否容易被放大”的风险评分。分数越高，越说明来源复杂或波动传导风险高，系统会更保守。

影响项加分规则为什么影响风险
来源 MSKU 数4-8 个 +10；9-20 个 +20；超过 20 个 +30。一个 SKU 被越多 MSKU 拉动，任一 MSKU 的误差都会叠加到 SKU 需求上。
来源 Type 数2-3 个 +8；4 个及以上 +15。跨类目来源越多，需求口径越混杂，预测传导更不稳定。
低稳定/样本不足来源占比占比 20%-50% +10；超过 50% +20。来源 MSKU 本身不稳定时，映射到 SKU 后更容易产生牛鞭效应。
人工调整来源占比占比 20%-50% +10；超过 50% +20。如果来源 MSKU 很多已经被人工调整，说明原始模型不确定性更高。
月度增长过快任一月份环比增长超过 30% +10；超过 60% +20。SKU 需求突然跳涨，容易导致过量备货或误判真实需求。
活动/清仓来源存在活动或清仓来源 +10。活动和清仓通常不可持续，不应完全外推到常规需求。
迭代转入来源存在转入 +8；转入来源达到 3 个及以上 +12。清库/停售 SKU 的需求转到新 SKU，会增加聚合风险和识别难度。
风险分 = 来源MSKU分 + 来源Type分 + 低稳定占比分 + 人工调整占比分 + 月增幅分 + 活动清仓分 + 迭代转入分 系统平衡版 = 映射版本 × 风险平衡系数 人工终版 = 有人工直改则取直改；否则 系统平衡版 × SKU系数 × 月度系数 + 加减量

六、人工调整规则

MSKU 端

  • 优先在 Type 目标层做调整，适合“某个类目整体变强/变弱”的场景。
  • 只有个别产品有特殊判断时，再进入 SKU 预测明细做单品月份直改。
  • 调整后建议填写原因，便于后续复盘为什么和公式预测不同。

SKU 端

  • 默认不要频繁改 SKU，先看来源和风险原因，确认系统压制是否合理。
  • 如果运营明确知道某个 SKU 需要备多或备少，可以编辑人工终版月份值。
  • 人工终版会覆盖系统平衡版，因此调整前要确认不是映射缺失或迭代关系错误导致的异常。
不建议直接为了“让数字变好看”而修改终版。人工调整应对应明确业务原因，例如促销、渠道策略、清仓转入、组合售卖变化或异常订单判断。

八、常见问题

1. 为什么 MSKU 系统已经有 3000 多条，但 SKU 系统生成数量少？

SKU 系统只会生成“有映射关系且通过数据诊断”的 SKU。如果 MSKU 缺映射，或清库/停售 SKU 缺少迭代 SKU，系统会阻止或减少生成结果。先看数据诊断，再补映射或状态表。

2. 为什么 SKU 系统的系统平衡版比映射版本低？

这是风险平衡逻辑。来源过多、跨 Type、低稳定来源占比高、环比增长过快、活动/清仓来源或迭代转入都会提高风险分，风险越高平衡系数越低。

3. 最终应该看哪个数字？

MSKU 端看“最终预测销量/最终预测销售额”；SKU 端看“人工终版”。如果没有人工修改，SKU 人工终版默认等于系统平衡版。

4. 什么时候需要人工调整？

当运营有明确业务信息，并且系统无法从历史销量或映射关系中自动识别时才调整，例如促销、清仓、本地仓策略、组合售卖变化、异常订单剔除等。

5. 保存 SKU 输入包云端前为什么会弹出数量确认？

这是防呆提醒。保存后会覆盖 SKU 系统读取的最新输入包，弹窗会显示本次保存 MSKU 条数和有效条数，避免误把测试包或旧缓存保存到云端。

6. 打开后没有数据，或者云端版本读取不到怎么办？

先回到系统入口页填写自己的白名单邮箱并点击“检查权限”。如果未通过白名单校验，系统可能无法读取云端数据，需要联系管理员补充白名单后再刷新重试。
