销售预测工作台说明文档

本文档写给运营使用，说明两个预测系统的操作顺序、预测口径、字段含义和结果核对方法。这里不包含 GitHub、云端脚本、权限配置等部署步骤。

返回系统入口 进入 MSKU 系统 进入 SKU 系统

一、整体用途

工作台分为两个独立系统：先用 MSKU 销售预测系统生成“销售侧预测”，再用 SKU 需求预测系统把 MSKU 预测拆解到真实 SKU 需求，并通过风险平衡降低牛鞭效应。

MSKU 销售预测系统 面向销售表现，按 MSKU、Type、价格、历史销量、季节性和人工目标生成未来 12 个月最终预测销量与销售额。
SKU 需求预测系统 面向供应需求，把 MSKU 最终预测按映射关系转成 SKU 需求，再结合风险评分、销售状态和人工修正得到 SKU 人工终版。
两套系统独立运行 MSKU 负责预测输入，SKU 负责需求拆解和风险平衡。两边不混在一个页面里，避免数据量过大导致卡顿。
核心原则：MSKU 预测看销售趋势，SKU 预测看真实需求和供应风险。运营调整应先确认销售预测，再进入 SKU 维度检查映射、迭代、风险和终版需求。

二、标准操作流程

Step 1进入 MSKU 系统，导入历史销量基础数据。
Step 2检查预测、目标涨幅、Type 汇总和趋势曲线。
Step 3确认后保存 SKU 输入包到云端。
Step 4进入 SKU 系统，导入最新 MSKU 预测包。
Step 5查看风险、来源、终版需求，必要时人工调整并导出。

MSKU 系统操作顺序

  1. 导入历史销量数据，确认 SKU 数、有效 SKU、价格、Type、历史月份读取正常。
  2. 查看月度销售额趋势、Type/类目汇总、趋势曲线和 SKU 预测明细，确认整体方向是否符合业务预期。
  3. 如需要设置目标增长，先在“月度目标分配”填写全局目标，点击“全局应用”生成 Type 建议。
  4. 确认 Type 建议后，点击“类目应用”，才会把 Type 涨幅写入 SKU 最终预测。
  5. 若个别 MSKU 需要特殊处理，可在 SKU 预测明细里编辑 12 个月终版预测并填写调整原因。
  6. 确认无误后，点击“保存 SKU输入包云端”，供 SKU 系统读取。

SKU 系统操作顺序

  1. 点击“导入最新MSKU预测包”，系统会读取 MSKU 端保存的最新 SKU 输入包。
  2. 系统自动加载内置 MSKU-SKU 映射关系和 SKU 销售状态表。
  3. 点击“生成SKU预测”，系统会检查缺映射、停售/清库缺迭代、迭代循环等问题。
  4. 生成成功后，先看数据诊断和异常 SKU 看板，再查看 SKU 预测总表。
  5. 点击 SKU 行的“详情/来源明细”，查看来源 MSKU 数、来源 Type、映射需求、风险和平衡系数。
  6. 如确需人工调整，点击编辑后修改任意月份的人工终版需求，填写备注并保存。

三、MSKU 预测口径

1. 基础预测链路

步骤说明运营关注点
历史销量读取读取 SKU、价格、Type、Status、负责人和历史月份销量。确认月份列识别正确，价格和 Type 不为空。
异常清洗对明显异常高值、异常低值和突然归零进行保守处理，避免单月极端值拉飞预测。关注 3 月旺季等真实高峰，不要把真实旺季误判为异常。
基础均值基于清洗后的历史销量形成较稳定的月均基准。这是预测的底座，不等于最终预测。
季节性 SI按 Type/类目读取 12 个月 SI 指数，让预测跟随品线季节趋势。缺 SI 的 Type 会提示补充。
稳定度与趋势结合 SKU 稳定层级、近期趋势、Type 趋势，控制预测波动幅度。稳定 SKU 更贴近模型，低稳定 SKU 更谨慎。
目标校准通过全局目标和 Type 目标调整未来月份，但必须点击应用才生效。未点击应用前只是建议或待生效值。
人工终版最终预测 = 公式预测 + 目标校准 + 活动/策略/人工修正。终版才是给后续 SKU 系统使用的口径。

2. 核心公式口径

清洗后销量 → 基础月均 → 结合 SI 季节性 → 结合稳定度/趋势 → 公式预测 公式预测 × Type目标系数 + 活动加量/人工直改 = MSKU最终预测 MSKU最终预测 × 价格 = 预估销售额

3. 月度目标分配

月度目标分配分为两层：先从全局目标拆到 Type 建议，再从 Type 目标写入 SKU 最终预测。这样可以避免输入一个全局涨幅后立刻重算几千条 SKU，降低卡顿和误操作风险。

按钮作用影响范围
全局应用按 Type 稳定度和趋势，把全局涨幅分配成 Type 建议涨幅。只影响 Type 建议值，不直接改 SKU。
类目应用把 Type 最终涨幅写入对应 SKU 的预测。刷新 SKU 明细、KPI、趋势条和销售额汇总。
清空人工覆盖清除 Type 人工覆盖值。需要再次点击类目应用才会影响最终预测。

四、SKU 预测口径

1. 三个版本的含义

版本怎么得到用途
映射版本MSKU 最终预测 × 映射用量，按最终 SKU 聚合。表示销售预测自然传导出来的 SKU 需求。
系统平衡版映射版本 × 风险平衡系数。用于抑制牛鞭效应，避免高风险 SKU 需求被过度放大。
人工终版默认等于系统平衡版；有人工系数、加减量或直改后，以人工规则为准。运营确认后的最终 SKU 需求。

2. 映射与迭代逻辑

SKU映射需求 = SUM(MSKU最终预测 × 映射用量) 如果多个 MSKU 指向同一个 SKU，会自动聚合。 如果 SKU 状态为清库/停售且有迭代 SKU，会把需求转入迭代 SKU 后再聚合。 如果清库/停售 SKU 有预测需求但没有迭代 SKU，系统会阻止生成并提示补全。
映射表缺失或停售/清库缺迭代，会直接影响 SKU 需求结果。遇到红色诊断时，优先补映射关系或 SKU 状态表，不建议绕过。

3. 风险评分与平衡系数

SKU 系统会根据来源复杂度、稳定性、人工调整、增长幅度、活动/清仓和迭代转入情况计算风险分。风险越高，系统平衡系数越低。

风险来源触发逻辑可能加分
来源 MSKU 过多来源 MSKU 达到 4、9、20 个以上。约 +10 / +20 / +30
跨 Type 来源同一 SKU 来源跨 2 个或 4 个以上 Type。约 +8 / +15
低稳定来源占比高低稳定或样本不足来源占比达到 20% 或 50%。约 +10 / +20
人工调整来源占比高来源 MSKU 中已有人工调整占比较高。约 +10 / +20
月度增长过快映射需求环比增长超过 30% 或 60%。约 +10 / +20
活动/清仓来源来源 MSKU 带活动或清仓标记。约 +10
迭代转入清库/停售 SKU 的需求转入当前 SKU。约 +8 / +12
风险等级风险分系统平衡系数解释
低< 251.00风险较低，不做压制。

五、关键字段解释

字段所在系统含义受什么影响
公式预测销量MSKU系统基于历史销量、SI、稳定度和趋势算出的模型预测。历史销量、异常清洗、SI、趋势和稳定度。
最终预测销量MSKU用于销售额汇总和传给 SKU 系统的最终销量。公式预测、Type目标、活动加量、人工直改。

六、人工调整规则

MSKU 端

  • 优先在 Type 目标层做调整，适合“某个类目整体变强/变弱”的场景。
  • 只有个别产品有特殊判断时，再进入 SKU 预测明细做单品月份直改。
  • 调整后建议填写原因，便于后续复盘为什么和公式预测不同。

SKU 端

  • 默认不要频繁改 SKU，先看来源和风险原因，确认系统压制是否合理。
  • 如果运营明确知道某个 SKU 需要备多或备少，可以编辑人工终版月份值。
  • 人工终版会覆盖系统平衡版，因此调整前要确认不是映射缺失或迭代关系错误导致的异常。
不建议直接为了“让数字变好看”而修改终版。人工调整应对应明确业务原因，例如促销、渠道策略、清仓转入、组合售卖变化或异常订单判断。

问题优先检查处理方式
MSKU 销售额趋势异常平或异常跳价格、Type目标、SI、人工直改。先看月度目标分配是否应用，再看单品是否被直改。
SKU 生成数量明显偏少MSKU-SKU 映射是否完整、是否加载了最新映射。先清空 SKU 本地缓存，再导入最新 MSKU 包；仍异常则补映射。

八、常见问题

1. 为什么 MSKU 系统已经有 3000 多条，但 SKU 系统生成数量少？

SKU 系统只会生成“有映射关系且通过数据诊断”的 SKU。如果 MSKU 缺映射，或清库/停售 SKU 缺少迭代 SKU，系统会阻止或减少生成结果。先看数据诊断，再补映射或状态表。

2. 为什么 SKU 系统的系统平衡版比映射版本低？

这是风险平衡逻辑。来源过多、跨 Type、低稳定来源占比高、环比增长过快、活动/清仓来源或迭代转入都会提高风险分，风险越高平衡系数越低。

3. 最终应该看哪个数字？

MSKU 端看“最终预测销量/最终预测销售额”；SKU 端看“人工终版”。如果没有人工修改，SKU 人工终版默认等于系统平衡版。

4. 什么时候需要人工调整？

当运营有明确业务信息，并且系统无法从历史销量或映射关系中自动识别时才调整，例如促销、清仓、本地仓策略、组合售卖变化、异常订单剔除等。

5. 保存 SKU 输入包云端前为什么会弹出数量确认？

这是防呆提醒。保存后会覆盖 SKU 系统读取的最新输入包，弹窗会显示本次保存 MSKU 条数和有效条数，避免误把测试包或旧缓存保存到云端。

6. 打开后没有数据，或者云端版本读取不到怎么办？

先回到系统入口页填写自己的白名单邮箱并点击“检查权限”。如果未通过白名单校验，系统可能无法读取云端数据，需要联系管理员补充白名单后再刷新重试。
